# Stale Python Bytecode Due to Tracked `.pyc` in Git

Use this when Python executes old behavior that doesn't match the current `.py` source — e.g. a method that was changed to return `dict` still returns `None` at runtime, causing `AttributeError: 'NoneType' object has no attribute 'get'`.

## Symptom

- The `.py` source on disk contains the fix, but Python behaves as if running the old version.
- An `AttributeError` on a method return value that should be the correct type now.
- `git diff` on the `.py` file shows no (or expected) changes — the source looks right.
- Restarting the process doesn't help.

## Diagnosis

### Step 1 — Check if the file was recently changed

```bash
git log --oneline -5 -- path/to/file.py
```

Look for a commit that changed the method/function in question. If the commit's change matches what "should" be happening but isn't, you have a bytecode staleness issue.

### Step 2 — Check if `.pyc` files are tracked in git

```bash
git ls-files '*.pyc' | head -5
```

If this returns `.pyc` file paths, they're tracked. This is the root cause: commits update `.py` files but leave old `.pyc` bytecode in the index.

### Step 3 — Check `.gitignore`

```bash
cat .gitignore
```

The typical culprit: `.gitignore` is missing `__pycache__/` and `*.pyc` entries. Without these, `git add .` or a broad commit captures compiled bytecode alongside source.

## Fix

### 1. Add to `.gitignore`

```
__pycache__/
*.pyc
```

### 2. Untrack all `.pyc` files from git

```bash
git ls-files '*.pyc' | xargs -r git rm --cached
```

This removes them from git's index without deleting the on-disk files (they'll be regenerated by Python on next import).

### 3. Clear local `__pycache__` directories

```bash
find . -type d -name __pycache__ -exec rm -rf {} + 2>/dev/null
```

### 4. Restart

Python will recompile fresh bytecode from the current `.py` sources on next import. The stale behavior is gone.

## Why this happens

Git tracks files, not modules. When a `.py` file is changed in a commit but the corresponding `.pyc` isn't regenerated and re-committed, checking out that commit writes the old bytecode to disk. Python's import system checks if the `.pyc` is stale by comparing its embedded source timestamp against the `.py` mtime — but git checkouts give both files the same (current) mtime, potentially defeating the staleness check. The old bytecode wins.

## If purging `__pycache__` doesn't fix it

Another cause of the same symptom (old code runs despite correct source) is a stale **installed copy** of the module in `site-packages/`. This happens when hatchling's `force-include` copies a `.py` file into the wheel at install time, and the source has been modified since. See `hatchling-force-include-stale` for diagnosis (check `module.__file__`) and fix (`uv sync --reinstall`).
